## Phase Boundary

Phase 23 은 `EdgeToLineDistanceMeasurement` 알고리즘(Datum-relative Y거리 계산)을 만들었으나 두 가지 갭을 남겼다:

1. **ROI 티칭 미배선** — measurement 가 측정에 쓰는 `Point_*` ROI 필드에 UI 드로잉 경로가 연결되지 않음. `CommitRectRoi` 는 `FAIConfig.ROI_*` 로만 쓴다.
2. **다점 치수 미지원 가정** — "1 FAI = 측정 1개 = 점 1개" 로 운용. SOP(A8.1 Z-Stopper, p.39~43)는 A1~A5 가 Datum B 선으로부터 점 1~N개까지의 Y거리를 측정하도록 요구한다.

Phase 23.1 은 이 갭을 닫는다:
- 드로잉한 Rectangle ROI 를 measurement별 `Point_*` 에 배선 (measurement 단위 ROI 티칭)
- 1개 FAI 가 N개 measurement(다점)를 갖는 유동 구조를 UI·INI 로 운용 가능하게 함
- `EdgeSelection` 결함(CO-23-01 #1)을 구조적으로 차단
- A1~A5 의 Datum→Y거리(mm) 측정을 실데이터 SIMUL 로 검증하고, Phase 23 의 보류된 UAT 를 흡수해 Phase 23 을 최종 마감

**범위 내:** ROI 측정별 티칭 배선 · 다점 치수(1 FAI = N measurement) UI/INI 운용 · EdgeSelection 구조적 차단 · A1~A5 Datum→Y(mm) 실측 검증 + CO-23-01/Phase 23 UAT 통합 재검증

**범위 외:** A6~A23 추가 (코드는 확장 지원하나 검증은 범위 밖) · B·C 시리즈 / X축 거리 측정 · Top Fixture #2+ / Bottom Fixture · 신규 측정 알고리즘 타입 · Polygon/Circle 형태의 Point ROI · EdgeToLineDistance 외 measurement 타입의 ROI 입력/EdgeSelection 변경

**요구사항:** Phase 23.1 은 별도 REQ-ID 없이 **ALG-01 carry-over** 를 완결한다 (Phase 23 = ALG-01 partial, 23.1 = ALG-01 미완 마감).

## Implementation Decisions

### ROI 측정별 티칭 배선

- **D-01:** 드로잉 ROI → measurement 배선 = **기존 Rect ROI 버튼 컨텍스트 재사용**. 트리에서 **Measurement 노드** 선택 시 Rect ROI commit 이 그 measurement 의 `Point_Row/Point_Col/Point_Phi/Point_Length1/Point_Length2` 로 쓰고, **FAI 노드** 선택 시 기존대로 `FAIConfig.ROI_*` 로 쓴다 (컨텍스트 분기). 신규 toolbar 버튼 추가 없음. 작동하는 analog = `CircleDiameterMeasurement` 의 `CommitCircleRoi`→`Circle_*` + `_editingCircleMeasurement` 상태 경로 — EdgeToLineDistance 는 `_editingMeasurement` 류 상태 + `CommitRectRoi` 분기로 동일 패턴을 Rectangle 에 적용.
- **D-02:** 배선 적용 범위 = **`EdgeToLineDistanceMeasurement` 만**. `PointToLineDistanceMeasurement`, `PointToPointDistanceMeasurement` 등 다른 Point ROI 보유 타입의 ROI 입력 경로는 무수정 (회귀 위험 최소, 23.1 범위 한정).
- **D-03:** 다점 FAI 티칭 중 ROI 시각화 = **FAI 노드 선택 시 그 FAI 의 모든 measurement Point ROI 동시 렌더**, Measurement 노드 선택 시 해당 ROI 강조. P1·P2 의 상대 위치를 보며 티칭 가능.

### 다점 치수 구조

- **D-04:** 다점 치수 = **"1 FAI = N measurement" 유동 구조**. FAI당 measurement 개수는 1/2/3+ 임의이며 **레시피 데이터로 결정**된다. 코드는 기존 `FAIConfig.Measurements` (`List<MeasurementBase>`) 를 그대로 N개로 운용하고, 특정 FAI 집합(A3/A4 또는 A1/A3/A4/A5)을 하드코딩하지 않는다. SOP 요약표의 "2x" 표기는 코드 분기 근거가 아니다 — 어느 FAI 가 몇 점인지는 사용자가 INI/UI 로 설정.
- **D-05:** measurement 추가/삭제 채널 = **UI 'Add Measurement' + INI 직접 편집 둘 다**. INI 는 기존 `[SHOT_{s}_FAI_{f}_MEAS_{m}]` 섹션 + `MeasurementCount` 패턴 그대로. UI Add 는 기존 InspectionListView 의 FAI CRUD 패턴을 measurement 레벨로 확장. Remove 도 대칭 제공 (기존 FAI Remove 패턴 따름).
- **D-06:** FAI 결과 표시 = **FAI 행은 OK/NG 종합 판정만**. 기존 `faiAllPass` (모든 measurement OK → FAI OK) 동작 유지. P1·P2 등 점별 측정값은 각 **Measurement 트리 노드**에서 확인. FAI 행에 다점 값 병기 안 함 — 기존 트리 2계층 표시 그대로.
- **D-07:** 다점 공차 = **measurement 별 독립 입력**. 각 measurement 의 `MeasurementBase` Nominal/Tolerance 필드를 PropertyGrid 에서 각각 입력. FAI 레벨 공유 nominal 도입 안 함. SOP "2x18.500" 같은 항목도 P1·P2 measurement 각각 nominal/tolerance 입력. 코드 변경 0.

### EdgeSelection 처리

- **D-08:** `EdgeToLineDistanceMeasurement.EdgeSelection` = **내부 `"All"` 고정**. line fit(`FitLineContourXld`)에 edge 점 ≥2 필요 → `"First"/"Last"`(1점)는 항상 측정 실패(CO-23-01 결함 #1, commit 898a96f 에서 default 만 교정됨). `TryExecute` 는 항상 명시적 `"All"` 을 `TryFitLine` 에 전달 (memory `feedback_halcon_measurepos_must_haves` 의 "EdgeSelection 명시" 요건 충족 — 코드 레벨 명시이지 UI 노출 의무 아님).
- **D-09:** `EdgeToLineDistanceMeasurement` 에 **`ICustomTypeDescriptor` 도입 → `EdgeSelection` 을 PropertyGrid 에서 숨김**. Phase 19 의 `DynamicPropertyHelper.FilterProperties` 정적 헬퍼 재사용. 잘못된 사용자 선택을 원천 차단.
- **D-10:** **별도 EdgeSelection 마이그레이션 로직 없음**. 레거시 INI 의 `EdgeSelection=First` 값은 로드되더라도 `TryExecute` 가 무시(`"All"` 고정)하므로 무해. INI 키 자체는 `ParamBase` 직렬화 호환을 위해 유지.
- **D-11:** EdgeSelection 변경 범위 = **`EdgeToLineDistanceMeasurement` 만**. 다른 measurement 타입의 EdgeSelection 동작/노출은 무수정.

### 검증

- **D-12:** 23.1 검증 = **실데이터 SIMUL UAT**. 사용자가 다점 geometry 가 보이는 Simul 이미지 + Datum CTH INI + A1~A5 FAI INI(다점 포함) 를 셋업한 뒤 SIMUL_MODE 실행. ROI 측정별 티칭 / 다점 측정 / A1~A5 측정값을 직접 확인. 해당 UAT plan 은 `autonomous: false`.
- **D-13:** 측정값 정확도 판정 = **SOP nominal/tolerance 대조**. A1~A5 측정값과 OK/NG 판정이 SOP 공차와 일치하는지 확인. SOP 기준값: A1 = 20.681 ±0.030 / A2 = 19.771 ±0.050 / A3 = 18.500 +0.050/-0.020 (2점) / A4 = 2.180 +0.020/-0.050 (2점) / A5 = 1.079 ±0.050.
- **D-14:** Phase 23 마감 = Phase 23 UAT 의 blocked 시나리오(SC#1~SC#4) + 23.1 신규 검증(ROI 티칭·다점)을 **`23.1-UAT.md` 하나로 통합 재검증**. 23.1 sign-off = Phase 23 도 최종 sign-off (Phase 23 의 `23-UAT.md` 는 23.1 로 superseded). SC#5(msbuild)는 Phase 23 에서 이미 PASS.

## Specific Ideas

- SOP 요약표는 "2x" 를 A3·A4 에만 표기하나 이전 메모리 분석(p.39~43)은 A1/A3/A4/A5 를 2점으로 기록 — **불일치는 D-04 로 해소**: 코드는 어느 FAI 가 몇 점인지 모르고, FAI당 measurement 개수는 레시피(INI/UI)로 유동 결정. 정확한 집합은 사용자가 셋업 시 입력.

- Phase 24 (검사 워크플로우 e2e) 는 A시리즈 동작에 의존 — ROADMAP 의 Phase 24 `Depends on` 을 `Phase 23` → `Phase 23.1` 로 갱신 검토 권장 (roadmap 작업, 23.1 범위 밖).
