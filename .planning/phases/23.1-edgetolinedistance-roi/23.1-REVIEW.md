## Warnings

### WR-01: `datumOriginInjected` 판별 조건이 유효 주입을 오탐할 수 있음

**File:** `WPF_Example/Custom/Sequence/Inspection/Measurements/EdgeToLineDistanceMeasurement.cs:138`

**Issue:**
```csharp
bool datumOriginInjected = (DatumOriginRow != 0.0 || DatumOriginCol != 0.0);
```
Datum 교점이 이미지 원점(0,0)에 실제로 위치할 때 — 예를 들어 Datum ROI가 이미지 좌상단 영역에 설정되거나 오정렬된 경우 — 이 조건은 `false`를 반환하여 `projection_pl` 정사영 경로가 아닌 레거시 `AffineTransPoint2d` 폴백 경로로 빠진다. 폴백 경로는 `MeasureAxis = "X"` 설정을 완전히 무시하고 Y 기준 부호 반전만 수행하므로, 사용자가 인식하지 못하는 상태에서 측정값이 다른 공식으로 계산된다.

주석에도 "미주입 시 0"이라 명시되어 있어 0이 "주입되지 않음"의 센티넬 값으로 사용되고 있음을 알 수 있지만, 실제 Datum 좌표가 0인 경우와 구별하지 못한다.

**Fix:**
별도의 주입 완료 플래그를 사용하거나 `DatumAngleRad != 0` 조건을 결합하여 실제 주입 여부를 판단한다. 가장 단순한 방법은 전용 플래그 필드를 추가하는 것이다.

```csharp
// IDatumOriginConsumer 계약에 따라 Action_FAIMeasurement가 주입 완료 시 true로 설정.
[System.ComponentModel.Browsable(false)]
[Newtonsoft.Json.JsonIgnore]
public bool IsDatumOriginInjected { get; set; } = false;

// TryExecute 내부
bool datumOriginInjected = IsDatumOriginInjected;
```

Action_FAIMeasurement가 `DatumOriginRow/Col`을 쓴 직후 `IsDatumOriginInjected = true`를 추가로 설정한다.

### WR-02: `projection_pl` 성공 여부와 무관하게 `resultValue` 계산이 진행됨

**File:** `WPF_Example/Custom/Sequence/Inspection/Measurements/EdgeToLineDistanceMeasurement.cs:196-214`

**Issue:**
```csharp
try {
    HOperatorSet.ProjectionPl(...);
    footRow = prRow.D;
    footCol = prCol.D;
    footOk = true;
}
catch {
    // projection 실패 시 foot=에지점 유지 → 측정값 0, FAI-DistLine skip
}
double signedPx; // catch 블록을 지나도 이 코드는 항상 실행
if (measureX) {
    ...
    signedPx = (pRow - footRow) * cosT - (pCol - footCol) * sinT;
```

`footOk = false`인 경우(`ProjectionPl` 예외 발생 시) `foot{Row,Col}`은 초기값 `(pRow, pCol)` 그대로이다. 이 상태에서 `signedPx = (pRow - footRow) * ... + (pCol - footCol) * ...`는 항등적으로 `0`이 되고, `resultValue = 0`이 반환된다. 그런데 주석에는 "측정값 0"이라 적혀 있어 의도적인 것처럼 보이지만, FAI 판정에서는 `NominalValue = 0`이 아닌 한 이 측정값이 OK/NG를 결정하므로 사용자 혼란을 야기한다.

`footOk = false`일 때는 `resultValue = 0`이 아니라 `TryExecute`가 `false`를 반환하고 `error`를 설정하는 것이 올바른 계약이다.

**Fix:**
```csharp
catch {
    // projection 실패 시 측정 불가로 처리
    error = "ProjectionPl failed";
    return false;
}
```
또는 `footOk` 체크 후 조기 종료:
```csharp
if (!footOk) {
    error = "datum projection failed";
    return false;
}
```

### WR-03: `CommitRectRoi`의 Measurement 분기에서 `_editingMeasurementFaiName`가 비어 있을 때 하이라이트 ROI ID가 부모 FAI 전체를 가리킴

**File:** `WPF_Example/UI/ContentItem/MainView.xaml.cs:1805-1810`

**Issue:**
```csharp
string measSelId = _editingMeasurementFaiName;
if (string.IsNullOrEmpty(measSelId))
    measSelId = FindFaiNameContainingMeasurement(_editingMeasurement);
FAIConfig anchorFaiForCommit = FindFAIByName(measSelId);
var measRois = CollectShotRois(anchorFaiForCommit);
halconViewer.UpdateDisplayState(measRois, measSelId, null, null);
```

`measSelId`는 FAI 이름이다. 그런데 `UpdateDisplayState` 두 번째 인자는 `_selectedRoiId`로, `RoiDefinition.Id`와 정확히 일치해야 해당 ROI가 노란색으로 하이라이트된다. Measurement의 Point ROI는 `RoiDefinition.Id = faiName + "_" + measName` 복합 ID이므로, FAI 이름만 넘기면 Point ROI가 노란색이 아닌 초록색으로만 표시된다 — 커밋 직후 어떤 ROI가 방금 그려진 것인지 사용자가 구별하지 못한다.

이 문제는 `ArcLineIntersect` 순차 드로잉에서도 동일하게 발생한다.

**Fix:**
```csharp
string measName = _editingMeasurement.MeasurementName;
if (string.IsNullOrEmpty(measName)) measName = _editingMeasurement.TypeName;
string highlightId = measSelId + "_" + measName; // Point ROI composite ID
// ArcLineIntersect 마지막 ROI는 별도 suffix
halconViewer.UpdateDisplayState(measRois, highlightId, null, null);
```

### WR-04: `MeasureAxis="X"` + `DatumAngle2Rad=0` 폴백에서 부호 계산이 의도한 "+X 오른쪽 양수"와 불일치

**File:** `WPF_Example/Custom/Sequence/Inspection/Measurements/EdgeToLineDistanceMeasurement.cs:199-208`

**Issue:**
```csharp
if (measureX) {
    if (useAngle2) { // 실 수직 기준선
        signedPx = (pRow - footRow) * cosT - (pCol - footCol) * sinT;
    }
    else { // 폴백: (sinθ, cosθ) 공식 (datum x축 방향 성분)
        signedPx = (pRow - footRow) * sinT + (pCol - footCol) * cosT;
    }
}
```

폴백 경로(`useAngle2 = false`)의 `signedPx` 공식은 "datum x축 방향 성분"이라고 주석되어 있다. datum x축 방향벡터가 `(sinθ, cosθ)`(이미지 row/col 좌표계, θ=수평선 각도)이므로, 에지점과 projection foot 사이의 변위를 x축 방향으로 내적한 값이다. 그런데 "+X 오른쪽 양수" 설계(D-02 대칭)를 위해서는 datum 수직선의 **법선 방향** 성분이어야 하며, 수평선 방향벡터와 수직선의 법선은 방향이 다르다.

이 오류는 `DatumAngle2Rad`가 실제로 주입되는 경우(`useAngle2=true`)에는 발생하지 않으며, 2차 기준선 미지원 Datum 타입에서 `MeasureAxis="X"`를 사용할 때만 재현된다. 현재 carry-over로 명시된 설계 결정이지만, 코드가 "carry-over" 상태임을 런타임에서 감지할 방법이 없어 사용자가 잘못된 값을 OK로 수신할 수 있다.

**Fix:**
단기: 이 분기에 진입했음을 `error` 출력 없이라도 로그에 남긴다.
장기: `DatumAngle2Rad == 0`이고 `MeasureAxis = "X"`인 경우 `TryExecute`를 `false`로 반환하고 `error = "MeasureAxis=X requires DatumAngle2Rad (carry-over)"` 메시지를 제공한다. 사용자가 잘못 설정한 측정이 '—' 표시되어 검출 우선순위를 받는다.

### IN-03: `InspectionListView`의 Measurement 노드 `btn_rectRoi` 활성화 로직이 `ENodeType.Measurement` 분기에만 위치

**File:** `WPF_Example/UI/ControlItem/InspectionListView.xaml.cs:533-543`

**Issue:**
`btn_rectRoi.IsEnabled = isRectRoiType`는 `ENodeType.Measurement` 분기에서만 설정된다. 다른 노드(FAI, Shot, Datum, Sequence)가 선택되면 핸들러 상단의 `btn_rectRoi.IsEnabled = false` 기본 리셋이 없으므로, 이전 Measurement 선택에서 `true`로 설정된 상태가 유지된다. 현재 코드 라인 460 근처를 보면 `btn_circleRoi.IsEnabled = false`, `btn_teachDatum.IsEnabled = false`는 핸들러 시작 시 명시적으로 `false`로 리셋되지만 `btn_rectRoi`는 리셋이 없다.

이로 인해 Measurement 노드 → FAI 노드 순서로 클릭하면 `btn_rectRoi`가 `IsEnabled=true` 상태로 남아 있고, 클릭 시 `_editingMeasurement`가 이미 ExitCanvasMode로 null이 됐기 때문에 FAI Rect ROI 드로잉 경로로 분기된다. 직접적인 버그는 아니지만 사용자 혼란 및 의도치 않은 ROI 덮어쓰기로 이어질 수 있다.

**Fix:**
`InspectionList_SelectionChanged` 핸들러 상단의 버튼 초기화 블록에 추가한다.
```csharp
if (mParentWindow != null && mParentWindow.mainView != null) {
    mParentWindow.mainView.btn_circleRoi.IsEnabled = false;
    mParentWindow.mainView.btn_teachDatum.IsEnabled = false;
    mParentWindow.mainView.btn_rectRoi.IsEnabled = false; // 추가
}
```
