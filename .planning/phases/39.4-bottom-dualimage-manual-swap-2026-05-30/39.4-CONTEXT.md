# Phase 39.4 CONTEXT — Bottom DualImage 수동 swap UX 재설계

## Phase Boundary

DualImage Measurement (`DualImageEdgeDistanceMeasurement`) 의 가로/세로 **양측 모두 Measurement 단위 명시 경로** 로 재설계. 가로축 = `TeachingImagePath_Horizontal` 신규 필드 + 기존 `TeachingImagePath_Vertical` 유지. ShotConfig 공통 이미지 의존을 fallback 으로 강등.

작업 단위 = Bottom DualImage Measurement 한 종류 집중. Datum DualImage (Phase 36/37) 일관화는 별도 phase 로 이관.

## Origin (CO-39.3-01)

Phase 39.3 PARTIAL_SIGNED_OFF Test 2 (Swap UX) PASS 후 사용자 발견:
- Shot 이미지는 동일 Shot 의 다른 FAI 와 공유되는 공통 자원.
- 그러나 Phase 39.3 Horizontal swap 분기 (Plan 02 Task 02-04, `BtnSwapHorizontal_Click`) + 검사 분기 (`Action_FAIMeasurement.TryGrabOrLoadFaiDualImages`, L403 `pathA = ShotParam.SimulImagePath`) 가 ShotConfig 이미지를 DualImage Measurement 의 "가로축 티칭 이미지" 로 단독 점유.
- 결과 = 작업자가 "이 이미지가 이 Measurement 의 가로축 티칭 이미지인지, 다른 FAI 와 공유되는 공통 Shot 이미지인지" 즉각 인지 불가.
- 실 카메라 grab 환경 (POC 시연 기준) 에서 혼동 심화 예상.

## 4 Locked Decisions (D-G1 ~ D-G4)

### D-G1: Fallback 정책 = ShotConfig fallback (회귀 0)

**결정:** `TeachingImagePath_Horizontal` 이 빈 문자열이거나 파일 부재 시 → `ShotParam.SimulImagePath` 로 fallback (Phase 39.2/39.3 baseline 동작 그대로 보존).

**Why:**
- 기존 INI 레시피 (TeachingImagePath_Horizontal 부재) 자동 호환 — 신규 필드 default `""` 자동 채움 + fallback (a) = byte-identical 회귀 0
- 점진적 마이그레이션 가능 — 작업자가 안 설정해도 검사 계속 동작, 필요 시 후속 설정
- INI 호환 결정과 통합 (별도 Open Question 불필요)

**구현 영향:**
- `Action_FAIMeasurement.TryGrabOrLoadFaiDualImages` L403 분기 교체:
  ```
  string pathA = string.IsNullOrEmpty(dualMeas.TeachingImagePath_Horizontal) || !File.Exists(dualMeas.TeachingImagePath_Horizontal)
      ? ShotParam.SimulImagePath
      : dualMeas.TeachingImagePath_Horizontal;
  ```
- `MainView.BtnSwapHorizontal_Click` Measurement 분기에 동일 fallback 로직

### D-G2: Datum DualImage 일괄 적용 = 별도 phase (39.4 = Measurement 만 집중)

**결정:** Phase 36/37 의 Datum DualImage (`DatumConfig.TeachingImagePath` / `TeachingImagePath_Vertical`) 도 동일 attribute 패턴 적용은 **이번 phase 범위 밖**. 별도 후속 phase (예: 39.5 또는 v1.3) 로 이관.

**Why:**
- 39.4 의 Anti-Goal "Datum DualImage 회귀 0" 와 일관 — 회귀 표면 격리
- Datum 측은 Phase 36/37 sign-off 완료 상태 → 검증된 코드 건드릴 이유 0
- 39.4 = Measurement 한 종류 집중으로 UAT 부담 최소화

**이연:** Deferred Ideas 섹션에 명시 → 후속 phase 후보.

### D-G3: PropertyGrid 라벨 = [DisplayName] + [Category] 조합

**결정:** 가로/세로 이미지 경로 셀에 `[DisplayName("가로축 티칭 이미지")]` + `[DisplayName("세로축 티칭 이미지")]` 추가 + 기존 `[Category("Image|DualImage")]` 그룹 유지.

**Why:**
- 작업자가 셀 헤더만 보고도 즉각 인지 (한글화)
- `[Category]` 그룹 유지로 PropertyGrid 그룹화 동작 보존 (Phase 39.3 D-G3 baseline 회귀 0)
- PropertyTools 3.1.0 의 `DisplayName` 지원 확인 필요 (research-phase / plan-phase 에서 lock)

**구현 영향:**
- `DualImageEdgeDistanceMeasurement.cs` L29 `TeachingImagePath_Vertical` 위에 `[DisplayName("세로축 티칭 이미지")]` 추가
- 신규 `TeachingImagePath_Horizontal` 에 `[DisplayName("가로축 티칭 이미지")]` + `[Category("Image|DualImage")]` + `[InputFilePath]` + `[AutoUpdateText]` 4종 attribute

**Open question (research-phase 에서 lock):**
- PropertyTools 의 `[DisplayName]` (PropertyTools.DataAnnotations) vs `System.ComponentModel.DisplayName` — 어느 namespace 가 PropertyGrid 셀 헤더에 실제 반영되는지 확인. Fallback = `[Description]` 만 적용 (tooltip 한글화).

### D-G4: Swap UX 하이라이트 = 배지 라벨에 소스 명시

**결정:** `BtnSwapHorizontal_Click` 호출 시 fallback 동작 여부에 따라 배지 텍스트 동적 변경:
- `TeachingImagePath_Horizontal` 명시 + 파일 존재 → 배지 = `"가로축 (Measurement)"`
- 빈 문자열/파일 부재 → 배지 = `"가로축 (Shot fallback)"`

세로축 (`btn_swapVertical`) 은 항상 Measurement 단위 경로이므로 변경 0 → 배지 = `"세로축"` (또는 일관성 위해 `"세로축 (Measurement)"`).

**Why:**
- Swap 토글 누를 때마다 즉시 인지 — 추가 클릭 부담 0
- 다이얼로그/아이콘 같은 추가 UX 복잡도 없음
- Phase 39.3 D-G2 baseline (배지 = 가로축/세로축) 위에 텍스트 보강만, 색상/Visibility 변경 0

**구현 영향:**
- `MainView.UpdateImageSourceBadge` 또는 `BtnSwap*_Click` 안에서 배지 텍스트 동적 set
- Measurement 가 selected 일 때만 fallback 분기 적용 (Datum DualImage 는 기존 텍스트 보존)
