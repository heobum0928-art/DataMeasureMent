<domain>
## Phase Boundary

앱 더블클릭 → 창 표시까지 **18~20초 흰 화면**(체감 기동 정지)을 제거/마스킹하여 POC 시연 체감 속도를 개선한다. Phase 43에서 `[STARTUP] Total Initialize`는 ~579ms로 단축됐으나, 실제 흰 화면은 18~20초 — 즉 지연의 지배 구간은 `Initialize()` **밖**에 있다.

**고정 범위(ROADMAP CO-43-01):** 측정 알고리즘/검사 흐름 변경 없음. `SystemHandler.Initialize` 동작 의미 불변(순서 재배치만 허용). Phase 43 READY 마커 의미 유지. SIMUL 기준 검증.
</domain>

<analysis>
## 핵심 분석 (논의 진입 전 공유)

- **흰 화면의 구조적 원인:** `App.Application_Startup`(App.xaml:7 Startup 이벤트)가 `new MainWindow()` → `view.Show()` 순. `MainWindow` 생성자가 **`Show()` 이전에** 모든 무거운 작업을 끝낸다: `Initialize()`(MainWindow.xaml.cs:78, ~579ms) + `InitializeComponent()`(:81, 전체 MDI XAML inflation) + 이벤트 와이어링. Show 전까지 아무것도 paint 안 됨 = 흰 화면.
- **18~20초 ÷ Initialize 579ms** → 나머지 ~18초는 (a) process 콜드 JIT(Debug 빌드, NGEN 없음), (b) Halcon/OpenCV/카메라 SDK 네이티브 DLL 로딩, (c) `InitializeComponent` 대형 XAML inflation 의 합. **정확한 분해는 계측해야 안다** — 현 `[STARTUP]` 마커는 Initialize 내부만 덮음.
- **스플래시 부재:** App.xaml 에 `StartupUri`/`SplashScreen` 없음(Startup 이벤트 핸들러만). 즉시 시각 피드백 수단이 전무.
- **마스킹 vs 단축:** 콜드 JIT/DLL 로딩/XAML inflation 은 실제로 단축하기 어렵거나(코드 위험 큼) Release/NGEN 영역. 반면 **흰 화면을 즉시 스플래시로 가리는 것**은 비용 위치와 무관하게 저위험·즉효 — POC 체감 속도의 80/20.
</analysis>

<decisions>
## Implementation Decisions

### 마스킹 메커니즘 — GA-1
- **D-01:** **WPF 네이티브 `SplashScreen` 클래스**(정적 PNG, 관리 UI/JIT **이전**에 표시)로 흰 화면을 가린다. 별도 WPF `Window` 로딩 화면은 그 자체가 JIT+inflation 비용을 유발해 콜드스타트 마스킹엔 부적합 → 기각.
- **D-02:** **수동 제어**로 첫 실제 paint까지 유지: `App.OnStartup`/`Application_Startup` 진입 즉시 `new SplashScreen("스플래시.png").Show(false)`(autoClose=false) → `MainWindow.ContentRendered`(첫 렌더) 시 `splash.Close(0.3s fade)`. auto-close 는 MainWindow 생성 전 너무 일찍 닫혀 흰 화면 재노출 → 금지.

### 계측 우선순위 — GA-2
- **D-03:** 마스킹 **전에** 흰 화면 구간을 분해하는 계측을 먼저 추가한다(Plan Task 1). 마커 구간: (a) process→`App` 정적 진입, (b) `MainWindow` ctor 진입, (c) `Initialize()`, (d) `InitializeComponent()`, (e) ctor 종료→`Show()`→`ContentRendered`(첫 paint). App 레벨 정적 `Stopwatch`로 절대시각 기록, 기존 `[STARTUP]` 로그(`Logging.PrintLog((int)ELogType.Trace, "[STARTUP] ...")`) 형식 연계. 어느 구간이 18~20초를 먹는지 사용자 실행 로그로 수치화 → 후속 async 판단 근거.

### 스플래시 자산 — GA-3
- **D-04:** 정적 PNG 1개를 프로젝트에 추가(Build Action=Resource 또는 Content, 경로는 planner 확정). 내용 = 프로그램명("DataMeasurement") + "기동 중..." 정적 텍스트(스피너 등 애니메이션은 정적 PNG라 불가). 기존 `WPF_Example/Camera_DDA.ico` 브랜드 참고.

### async 리팩토링 범위 — GA-4
- **D-05:** 본 phase 1차 목표 = **계측 + 마스킹(저위험)**. `Initialize()`/`InitializeComponent()`의 실제 비동기 재구성은 계측 결과가 지배 구간을 명확히 가리킬 때만 plan에서 판단하되 **기본은 보수적(out-of-scope, 후속 phase 후보)**. 측정 readiness(첫 `$TEST` 수용, Phase 43 `[STARTUP] READY` 마커 의미)는 절대 깨지 않는다.

### 검증 + 회귀 안전 범위 — GA-5
- **D-06:** UAT(SIMUL) = (a) 기동 즉시(≤1s) 스플래시 표시 → 흰 화면 체감 제거, (b) 흰 화면 구간 분해 수치 로그 확보(지배 구간 입증), (c) 회귀 0: 기동 후 첫 로그인 / 첫 `$TEST` 검사 / MainView 정상, READY 마커 의미 유지.

### Claude's Discretion
- 스플래시 닫기 트리거(`MainWindow.ContentRendered` vs `Loaded`) 및 fade 시간 — D-02 계약(첫 실제 paint까지 유지) 만족 범위 내 재량.
- 계측 마커의 정확한 코드 삽입 위치(App/MainWindow) 및 로그 prefix(`[STARTUP]` 확장).
- 스플래시 PNG 디자인 디테일(레이아웃/색) — D-04 내용 충족 시 재량.
</decisions>
