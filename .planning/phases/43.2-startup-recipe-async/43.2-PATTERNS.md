private void Window_ContentRendered_LoadRecipe(object sender, EventArgs e) {
    this.ContentRendered -= Window_ContentRendered_LoadRecipe; // 1회 실행 후 제거

private readonly Thread _recipePreloadThread;
private volatile bool _isRecipeReady;

public void StartupLoadRecipeAsync(string recipeName) {
    if (_isRecipeReady || _recipePreloadThread?.IsAlive == true) return;
    var t = new Thread(() => StartupLoadRecipeWorker(recipeName)) {
        IsBackground = true,
        Name = "RecipePreload",
        Priority = ThreadPriority.BelowNormal

private void StartupLoadRecipeWorker(string recipeName) {

        }

    catch (Exception ex) {
        Logging.PrintErrLog((int)ELogType.Error, "[RECIPE] Async load failed: " + ex.ToString());

    finally {
        _isRecipeReady = true;
    }
}

## 스레드 안전성 분석 요약

### InspectionRecipeManager.Load() 호출 경로 (백그라운드에서 안전한가?)

`LoadRecipe` → `LoadFromIni` → `TryLoadNewFormat` → `RecipeManager.Load(loadFile)` → `LoadPhase6Format` → `ClearShots()` + `Shots.Add()`

**위험 지점:**
1. `InspectionRecipeManager.Shots` — `List<ShotConfig>`, 스레드 비안전.
   - 백그라운드 로드 중 UI 스레드에서 `Shots` 읽기 충돌 가능.
   - 현재 기동 시 UI 스레드만 접근하므로, **ContentRendered 이후 Dispatcher.Background 방식**이면 여전히 UI 스레드 단일 실행 → 안전.
   - 별도 Thread 방식이면 `lock` 필요.

2. `InspectionRecipeManager.ClearShots()` 내부 `MessageBox.Show` 포함 (`Load()` 의 `CustomMessageBox.Show`) — 백그라운드 Thread에서 UI 호출 시 크래시.
   - `LoadPhase6Format` 내 `CustomMessageBox.Show` (line 239-243) → 백그라운드 Thread 실행 불가.
   - 따라서 **별도 Thread + LoadRecipe 직접 호출은 불가**. Dispatcher 경유 필수.

3. `SequenceHandler.LoadFromIni` → `pSetting.CurrentRecipeName = name` — SystemSetting 의 `CurrentRecipeName` setter 가 스레드 안전한지 확인 필요.

**결론 — 안전한 방식:**
- `Dispatcher.BeginInvoke(DispatcherPriority.Background, LoadRecipe)` — UI 스레드 실행, ContentRendered 이후 지연.
- CustomMessageBox, UI 컨트롤 접근 문제 없음.
- OnRecipeChanged 이벤트 핸들러 Dispatcher 래핑 불필요 (이미 UI 스레드).
- 단, `DispatcherPriority.Background` 는 렌더링 유휴 시 실행 → 실질 지연 약간 있을 수 있음.
