# Phase 57.1 Plan 08: TryFitLine 에지 trim 위치 정렬 + % 절사 Summary

`VisionAlgorithmService.TryFitLine` 의 EdgeTrimCount 처리를 "정렬 없이 누적(strip)순서 양끝 trimCount개 제거"에서 "위치축 정렬 후 양끝 각 N% 절사(trimmed fit)"로 교체 — 끝 노이즈를 robust 하게 제거하고 중간 점만 피팅.

## What Was Built

`TryFitLine` 의 trim 블록(기존 :146~155)을 다음 로직으로 교체:

1. **위치축 정렬**: `scanHorizontal`(LtoR/RtoL = 세로 에지 분포)이면 `allRows`, 아니면(TtoB/BtoT = 가로 에지 분포)이면 `allCols` 를 정렬 키로 사용. `sortKey.TupleSortIndex()` 로 오름차순 인덱스를 얻어 `allRows.TupleSelect(order)` / `allCols.TupleSelect(order)` 로 두 튜플을 쌍 유지하며 동시 재정렬.
2. **양끝 % 절사**: `trimCount` 를 양끝 각각 백분율로 재해석 — `trimPct` 를 0~49 로 clamp, `removeEach = (int)(edgeCount * trimPct / 100.0)`. 상위 removeEach + 하위 removeEach 제거.
3. **안전 가드**: `edgeCount >= 4` 이고 `removeEach > 0` 이며 `(edgeCount - 2*removeEach) >= 2` 일 때만 절사 적용 — 미달 시 trim skip(원본 유지)으로 남는 점 >= 2 보장.

이후 `edgeCount < 2` 검사, `GenContourPolygonXld`, `FitLineContourXld`, `collectedEdges` 누적은 무변경 — 정렬·절사된 `allRows`/`allCols` 를 그대로 소비하므로 overlay 마젠타 raw 에지점도 절사 후 점만 반영.

## How It Works (사용자용 쉬운 설명)

라인 피팅 시 검출된 에지점들이 예전에는 "검출된 순서대로" 앞뒤를 잘라내서, 점들이 위치상 뒤죽박죽 섞여 있으면 엉뚱한 점이 잘리고 이상치(끝부분 노이즈)는 남았습니다. 이제는 먼저 점들을 **실제 위치 순서로 줄을 세운 뒤**, 양쪽 끝에서 EdgeTrimCount **퍼센트**만큼씩 잘라내고 가운데 점들로만 직선을 맞춥니다. 끝쪽 노이즈가 깔끔히 제거되어 피팅선이 안정됩니다.

주의: EdgeTrimCount 의미가 "점 개수"에서 "양끝 각 퍼센트(%)"로 바뀌었습니다. 예: 20 = 상위 20% + 하위 20% 제거. 기존 레시피의 EdgeTrimCount 값을 재확인해야 합니다.

## Deviations from Plan

None - plan executed exactly as written. 가이드 코드의 인라인 clamp(`int trimPct = trimCount; if ...`)를 가독성을 위해 다중 라인으로 풀어썼으나 로직은 동일.

## Decisions Made

- EdgeTrimCount 를 "개수" → "양끝 각 백분율(%, 0~49 clamp)" 로 재해석.
- 정렬 키는 strip 분포 축(scanHorizontal → row, else → col).
- 기존 `edgeCount > 2*trimCount+1` 가드를 `edgeCount >= 4` + `(edgeCount - 2*removeEach) >= 2` 로 교체해 샘플 수 비례 % 라 들쭉날쭉 가드 문제 해소.
- HALCON 인스턴스 메서드(`TupleSortIndex()`, `TupleSelect(HTuple)`, `TupleSelectRange(start,end)`) 사용 — 기존 TryFitLine 의 `allRows.TupleSelectRange(...)` 패턴과 일치, 컴파일 검증됨.
