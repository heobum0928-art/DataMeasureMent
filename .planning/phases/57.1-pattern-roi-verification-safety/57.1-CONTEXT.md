<domain>
## Phase Boundary

Phase 54~57에서 구축한 Datum 패턴매칭 위치보정(ALIGN-01/02) + 패턴 ROI UX의 UAT 피드백 4항목을 검증·보강한다. 알고리즘 자체(매칭/transform 산출)는 변경 범위가 아니며, **(a) 적용 가시성 확인, (b) 회전 진단, (c) 시각화 렌더 안정화, (d) 버튼 안전장치**가 스코프다.

**In scope:**
1. Top/Bottom 패턴매칭 보정이 실제 적용되는지 육안 확인 가능하게 (cyan 보정 ROI 표시 보장)
2. gen_rectangle2 length1/length2 장축 + baseline 회전각이 90° 어긋나지 않는지 **진단**(코드 수정 아닌 확증/시각화/로그)
3. 패턴 ROI 시각화 렌더 조건 안정화 (어느 노드 선택에서도 일관 표시)
4. 패턴 ROI 버튼 비-Datum 노드 비활성화 + 알림 메시지박스 안전장치

**Out of scope:**
- 패턴매칭 엔진/transform 산출 로직 변경 (PatternMatchService 알고리즘 불변)
- length1/length2 swap 코드 수정 (아래 D-02 참조 — 회귀 위험으로 명시 금지)
- 신규 Datum 알고리즘 타입
</domain>

<decisions>
## Implementation Decisions

### D-01: Top/Bottom 패턴 보정 적용 = 구조상 이미 가능, 가시성만 보강
- Top/Side/Bottom 모두 동일 `InspectionSequence` → `Action_FAIMeasurement` → `TryComposeAlign()` 경로. 시퀀스별 차별 없음. per-Datum `IsPatternAlignEnabled`(기본 false)로 켬.
- 따라서 신규 배선 불필요. **목표 = 보정이 걸렸는지 육안 확인**. 보정 ROI cyan 박스가 Top/Bottom 결과 화면에도 표시되도록 보장(→ D-03 시각화 안정화와 동일 작업으로 해결).
- 검증: Top/Bottom Datum에 IsPatternAlignEnabled on → 검사 → cyan 보정 ROI가 보정된 위치에 표시.

### D-02: gen_rectangle2 회전 = analytic 방식 정상, length swap 금지 — 진단만
- `FAIEdgeMeasurementService.cs:51` 주석: smallest_rectangle2 의 phi 90° 모호성 회피 위해 **analytic 회전** 채택 = 중심점 transform 이동(`AffineTransPoint2d`) + 회전각을 phi에 가산(`roiPhi = ROI_Phi + Atan2(transform[3], transform[0])`), length1/length2 유지. 이는 **강체 회전상 정상**. AABB 계산(`:111-114`)도 회전 사각형 기준으로 정확.
- **🔒 LOCKED: length1/length2 를 swap 하는 코드 수정 금지.** swap 은 정상 강체회전을 깨뜨림(회귀). 사용자 우려("90° 뒤바뀜")의 진짜 원인 후보는 (a) baseline 회전각, (b) 티칭 시 length 매핑 직관 불일치이며, 둘 다 **진단/시각화/로그**로 확증한다.
- 진단 산출물 2가지:
  - (a) **회전각 확증 로그** — DatumPhase 에서 산출된 align 회전각(deg)과 적용 후 roiPhi(deg)를 Trace 로그로 남겨, baseline 각이 90° 어긋남 없이 부품 tilt 와 일치하는지 UAT 시 확인 가능하게.
  - (b) **length 장축 매핑 문서/시각화** — HALCON 규약(length1 = phi 방향 반장축, length2 = phi 수직 반장축)을 코드 주석 + 측정 ROI 시각화에 반영. 가능하면 결과 화면 측정 ROI 박스가 phi 방향(length1 축)을 구분 표시(예: length1 축 끝에 짧은 틱/다른 색)하여 어느 축이 장축인지 육안 식별.
- ※ PatternMatchService 의 match angle(`curAngleDeg`)은 거칠어 측정 미사용 — 정밀 θ 는 line-fit/2-패턴 baseline 이 담당(`TryBuildAlignRigid` thetaRad). 진단은 thetaRad 산출 경로(InspectionSequence/DatumFindingService 의 baseline 각)를 대상으로 한다.

### D-03: 패턴 ROI 시각화 렌더 조건 안정화
- 현상: cyan 패턴 ROI 는 `_resultDatumOverlays` 가 채워질 때만 렌더(MainResultViewerControl). 이 리스트는 Measurement/Shot/FAI 노드 렌더 경로에서만 채워지고, **Datum 노드 단독 선택/티칭 모드에선 비어 안 보임** → "보일 때/안 보일 때".
- 수정: Datum 노드 선택 시에도 해당 Datum(들)을 `SetResultDatumOverlays`(또는 동등 경로)로 전달해 패턴 ROI 오버레이가 채워지게 한다. Phase 57-05 에서 추가한 패턴 토글(`chk_overlayPattern`) 상태와 일관되게 동작해야 함.
- 회귀 가드: 기존 Measurement/Shot/FAI 노드의 패턴/Datum 오버레이 표시 회귀 0.

### D-04: 패턴 ROI 버튼 안전장치 (비-Datum 비활성화 + 알림)
- 현상: `패턴1 / 패턴2 / 패턴모델생성` 버튼이 Datum 노드 선택 시 활성화되지만, 다른 노드(Measurement/Shot 등)로 이동 시 **명시적 비활성화 안 됨** → 상시 클릭 가능.
- 수정 2가지:
  - (a) **비-Datum 노드 선택 시 3버튼 `IsEnabled = false`** — 트리 선택 핸들러(InspectionListView 선택 분기)에서 Datum 분기는 enable, 그 외 분기는 명시적 disable.
  - (b) **클릭 핸들러 가드** — 만약의 활성 상태로 클릭돼도, 선택 노드가 Datum 이 아니면 `CustomMessageBox` 로 "Datum 티칭 존을 먼저 선택하세요" 알림 후 return (early-out). 기존 프로젝트 알림 패턴(CustomMessageBox) 사용.

### Claude's Discretion
- cyan 오버레이를 Datum 노드 선택 시 채우는 구체적 메서드/시그니처 (기존 SetResultDatumOverlays 미러 vs 신규 경로)
- length1 축 시각화 표현(틱/색/라벨) 방식 — 기존 RenderResultRoiBoxes 패턴 내에서 최소 침습
- 회전각 로그의 ELogType(Trace 권장) 및 포맷 문자열
