### WR-01: `GetContourXld` — 다중 contour HObject 처리 오류 (첫 contour 좌표만 누적)

**File:** `WPF_Example/Custom/EthernetVision/AlignShapeMatchService.cs:534-553`

**Issue:**
`AffineTransContourXld` 는 입력 XLD 객체에 포함된 contour 수에 관계없이 출력(`movedContours`)으로 동일한 수의 contour를 담은 HObject 를 반환한다. 이어서 호출하는 `GetContourXld(movedContours, out contRows, out contCols)` 는 **첫 번째 contour 하나의 점만** 반환하는 HALCON 동작이다(복수 contour 객체에서 `get_contour_xld` 는 첫 XLD만 처리). Shape 모델 contour 가 여러 segment 로 구성된 경우(복잡한 윤곽) 두 번째 이후 segment 좌표는 누락된다. 시각화가 모델 윤곽 일부만 표시되어 "에지가 끊어 보임" 증상이 발생할 수 있다.

HALCON 올바른 패턴은 `CountObj`로 contour 수를 먼저 구하고 각 contour 별로 `SelectObj` + `GetContourXld`를 반복하는 것이다.

**Fix:**
```csharp
// movedContours 가 복수 XLD 를 포함할 수 있으므로 CountObj + SelectObj 루프
HTuple numContours = null;
try {
    HOperatorSet.CountObj(movedContours, out numContours);
    int contCount = numContours[0].I;
    for (int ci = 1; ci <= contCount; ci++) {
        HObject singleCont = null;
        HTuple cRows = null;
        HTuple cCols = null;
        try {
            HOperatorSet.SelectObj(movedContours, out singleCont, ci);
            HOperatorSet.GetContourXld(singleCont, out cRows, out cCols);
            int pts = cRows.Length;
            int stride = (pts > EDGE_CONTOUR_MAX_POINTS) ? pts / EDGE_CONTOUR_MAX_POINTS : 1;
            if (stride < 1) stride = 1;
            for (int i = 0; i < pts; i += stride) {
                outRows.Add(cRows[i].D);
                outCols.Add(cCols[i].D);
            }
        }
        finally {
            if (singleCont != null) { try { singleCont.Dispose(); } catch { } }
            if (cRows != null) { try { cRows.Dispose(); } catch { } }
            if (cCols != null) { try { cCols.Dispose(); } catch { } }
        }
    }
}
finally {
    if (numContours != null) { try { numContours.Dispose(); } catch { } }
}
```

기존 `contRows`/`contCols` HTuple 변수 선언과 `GetContourXld` 직접 호출 블록(라인 535~553)을 위 패턴으로 교체한다. `totalPts`/`stride` 루프는 루프 내부로 이동. finally 블록의 `contRows`/`contCols` Dispose 유지.

### WR-02: `EDGE_CONTOUR_MAX_POINTS` stride 다운샘플 — 두 번째 contour 추가 시 이미 누적된 점 `Clear()` 부작용

**File:** `WPF_Example/Custom/EthernetVision/AlignShapeMatchService.cs:558-562`

**Issue:**
`TryExtractDetectedContour` 실패 시 `outRows.Clear(); outCols.Clear()` 를 호출한다. 이 메서드는 `Run` 내부에서 두 패턴(`.shm` 1, 2)에 대해 순서대로 동일한 `result.EdgeContourRows`/`result.EdgeContourCols` 리스트를 누적(`Add`) 용도로 전달한다.

**두 번째 패턴(shmPath2) 추출 실패 시** catch 블록이 이미 첫 번째 패턴에서 성공적으로 누적된 점들까지 `Clear()` 로 삭제한다. 결과적으로 contour 가 전혀 없는 상태로 시각화되며, 에러 로그(`contour[2] 추출 실패`)와 실제 효과(전체 contour 소멸)가 일치하지 않아 디버깅이 어렵다.

**Fix:**
실패 시 해당 호출에서 추가된 점만 롤백하거나(복잡), 또는 `Clear` 를 호출하지 않고 에러 문자열만 반환하도록 변경한다. 시각화 목적이므로 부분 성공(patternA 성공 / patternB 실패)도 유효하게 처리하는 방식이 더 견고하다.

```csharp
// catch 블록 수정: 공유 리스트 Clear 제거. 호출자가 개별 실패 로그로 판단.
catch (Exception ex) {
    error = "TryExtractDetectedContour: " + ex.Message;
    // outRows.Clear(); outCols.Clear();  <-- 제거: 누적분 파괴 방지
    return false;
}
```

### WR-03: `BuildJsonPath` — `MODEL_SUFFIX_1` (`"_1"`) `Replace` 가 파일명 전체에서 치환됨

**File:** `WPF_Example/Custom/EthernetVision/AlignShapeMatchService.cs:115-118`

**Issue:**
```csharp
string withoutExt = Path.Combine(
    Path.GetDirectoryName(shm1),
    Path.GetFileNameWithoutExtension(shm1).Replace(MODEL_SUFFIX_1, ""));
```

`Path.GetFileNameWithoutExtension("Tray_1.shm")` → `"Tray_1"` → `.Replace("_1", "")` → `"Tray"` (의도 정상).

그러나 레시피명 또는 모드명에 `_1` 이 포함될 경우(예: 사용자가 레시피명을 `Shot_1` 로 지정하거나, 미래 모드 확장 시) 기대 외 치환이 발생한다. 현재 `"Tray"` / `"Bottom"` 고정값 상황에서는 실제 버그가 발생하지 않지만, 코드 자체는 파일명 전체에 `Replace` 를 적용하는 취약한 패턴이다.

**Fix:**
접미사 `_1` 이 파일명 **끝**에만 위치함을 보장하는 `TrimEnd` 패턴으로 변경한다.

```csharp
string stem = Path.GetFileNameWithoutExtension(shm1);
// MODEL_SUFFIX_1 은 파일명 끝에만 존재하므로 TrimEnd 패턴이 안전
if (stem.EndsWith(MODEL_SUFFIX_1, StringComparison.Ordinal)) {
    stem = stem.Substring(0, stem.Length - MODEL_SUFFIX_1.Length);
}
string withoutExt = Path.Combine(Path.GetDirectoryName(shm1), stem);
return withoutExt + REF_POSE_EXT;
```

### IN-02: `DatumConfig(this)` — Align 뷰에서 `ParamBase` 소유자를 UserControl 로 전달

**File:** `WPF_Example/Custom/UI/TrayVisionView.xaml.cs:307` / `WPF_Example/Custom/UI/BottomVisionView.xaml.cs:319`

**Issue:**
```csharp
var dc = new DatumConfig(this);
```
`ParamBase(object owner)` 생성자는 owner 를 INI 직렬화·`OwnerName` 산출에 사용한다. `OwnerName` getter 는 `ActionBase` 또는 `SequenceBase` 타입을 기대하며, 그 외는 `null` 을 반환한다. `UserControl` 인스턴스를 전달하면 `OwnerName=null` 이 된다.

이 `DatumConfig` 는 INI 직렬화 없이 시각화 전용으로만 사용되므로 런타임 오류나 데이터 손상은 없다. 다만 `DatumConfig` 를 `SetDatumFindResultOverlay` 에 전달했을 때 뷰어 내부가 `OwnerName` 을 참조하는 경로가 있다면 null 문자열로 렌더될 수 있다.

**Fix:**
시각화 전용 임시 객체이므로 `null` owner 를 명시적으로 전달하면 의도가 더 명확해진다.
```csharp
var dc = new DatumConfig(null);  // 시각화 전용, INI 직렬화 없음
```
또는 `DatumConfig` 를 직접 생성하지 않고 뷰어의 시각화 전용 오버로드(존재 시)를 사용한다.
