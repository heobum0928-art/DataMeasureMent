Phase 23.1 UAT 에서 사용자가 직접 확인한 버그를 수정한다 — EdgeToLineDistance 측정값이
datum 교점(intersection point) 기준 Y거리가 아니라 이미지 상단(row 0) 기준 거리로 나온다.

근본 원인: `DatumFindingService.TryFindDatum` 이 생성하는 `transform` 은
part-drift 보정 델타(`Translate(curOrigin - RefOrigin) ∘ Rotate(deltaAngle)`)이지
datum-상대 좌표계로의 변환 행렬이 아니다 (DatumFindingService.cs:155-157 TwoLineIntersect,
cs:307-309 CircleTwoHorizontal — 동일 패턴 확인 완료). 그런데
`EdgeToLineDistanceMeasurement.TryExecute` 는 이것을 좌표 변환으로 오용한다 —
`AffineTransPoint2d(datumTransform, pRow, pCol)` 결과 row 를 datum-상대 좌표로 간주하고
`resultValue = -tRow * pixelResolution` 로 계산한다. SIMUL 모드에서는 티칭 이미지 == 검사
이미지이므로 `curOrigin ≈ RefOrigin` → transform ≈ Identity → `tRow ≈ pRow`(에지의 raw
이미지 row) → 측정값이 이미지 row 0 기준 거리가 되어버린다.

진짜 datum 교점 좌표는 `DatumConfig.DetectedOriginRow` / `DetectedOriginCol` 에 있다
(`IntersectionLl` 결과를 `TryFindCircleTwoHorizontal` 등이 write-back, DatumFindingService.cs:312-313).
이 값은 측정에 전달되지 않고 있다 — `TryExecute` 는 `datumTransform` 만 받는다.

수정 방식: `MeasurementBase.TryExecute` 시그니처는 6개 측정 타입이 공유하므로 확장하지 않는다.
EdgeToLineDistance 만 datum-절대 좌표가 필요하고 나머지 5종은 rigid-transform-invariant 한
상대 측정이므로, datum 교점 좌표를 `EdgeToLineDistanceMeasurement` 의 public 프로퍼티로
받고 `Action_FAIMeasurement` 가 타입 분기로 주입한다 (blast radius 최소, 진단서 권장안 B).

Purpose: SIMUL UAT 에서 측정값이 SOP 도면과 정확히 일치하도록 — datum 교점 기준 측정 정합성 확보.
Output: 수정된 EdgeToLineDistanceMeasurement.cs(교점 좌표 프로퍼티 + 계산/overlay 수정),
